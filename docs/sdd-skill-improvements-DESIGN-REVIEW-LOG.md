# Spec Review Log: SDD Skill Improvements
Act 1 (brainstorming) complete — spec locked with user. MAX_ROUNDS=5.
## Round 1 — Codex
**Findings**

1. `AskUserQuestion` is not defined anywhere in the existing skills, which currently use plain prompts plus `EnterPlanMode`; this spec may add an invalid tool dependency.
Fix: Define the exact available interaction primitive, or phrase it as “present 2-3 options and wait for explicit user choice” without naming a nonexistent tool.

2. The modularization plan does not preserve the full existing control flow. The proposed `prd/steps/` omits explicit “read DESIGN.md” and “explore codebase”; `discover`/`ideate` step lists omit git dirty-state checks, plan-mode, slug approval, branch/worktree setup, session file handling, commit, and handoff.
Fix: Add a step mapping table from every current step to its new file/location before implementation.

3. “Pure structural refactor” is false. Moving instructions into step files changes skill loading behavior and increases the chance an agent only reads the index.
Fix: Require each thin `SKILL.md` to explicitly say “read and follow each referenced step file in order,” and make each step file mandatory, not optional context.

4. Step-file self-containment is underspecified. “States what it reads / asks / hands off” is not enough for resumability after context compaction, especially with session files and approved checkpoints.
Fix: Require each step file to include checkpoint invariants, resume behavior, required files, stop conditions, and approval gates.

5. The “propose defaults, then confirm” rule conflicts with existing one-question-at-a-time interview behavior in `discover` and with brainstorming’s upstream skill behavior.
Fix: Specify exactly which questions become multiple-choice defaults and which remain single focused open questions.

6. The spec names `design-brainstorm`’s “approach-proposal step,” but that step is delegated to `superpowers:brainstorming`; the local skill may not control the internal wording unless an override is added.
Fix: Add an explicit brainstorming override for proposed approaches, including the exact default-selection format.

7. The granularity choice for `plan` is ambiguous for `ISSUE-N.md` inputs. An issue is already a slice; asking “fewer/larger vs more/smaller” could reopen PRD-level decomposition instead of plan task sizing.
Fix: Define separate wording for `ISSUE-N.md`: choose implementation task granularity, not feature/issue granularity.

8. The `plan` change may fight `superpowers:writing-plans`, which owns the actual plan shape. The spec says to ask before plan writing but does not say how the chosen option is passed into the invoked skill.
Fix: Add a new override to `writing-plans` that injects the selected granularity as a hard planning constraint.

9. The PRD “what vs how” rule is too broad. Banning library/framework names, method names, and file paths from Problem/Solution/User Stories is sensible, but “Solution” often needs user-visible platform or integration names.
Fix: Distinguish implementation technology from user-facing product/integration names, with examples of allowed and disallowed wording.

10. The rule says PRD body may not contain code, but existing issue files are also generated by `prd` and still allow prototype snippets.
Fix: State whether issue files inherit the no-code/no-path rule; if not, explicitly say “PRD.md only, not ISSUE-N.md.”

11. “Implementation Decisions may still name modules/schemas/API contracts” undermines “no how” unless bounded. Module names and API contracts are implementation design, not stakeholder PRD content.
Fix: Rename that section or constrain it to externally relevant contracts only, with no internal module/file references.

12. No verification criteria exist for the refactor itself. Manual dry-run with a trivial feature will not catch missing step files, broken references, or output drift.
Fix: Add static verification: check all referenced step files exist, all old mandatory checkpoints appear somewhere, and output templates remain unchanged except intended wording.

13. The branch note says `feature/sdd-skill-improvements` is created, but the spec does not require implementers to verify they are on it before editing.
Fix: Add a pre-implementation check: confirm current branch and dirty state before changing skill files.

14. Scope says no change to design-brainstorm/design-adversarial/design-review trio beyond defaults rule, but Item 2 only names `design-brainstorm`; the trio wording implies all three may be affected.
Fix: Replace “trio” with the exact skill names affected, or explicitly say `design-adversarial` and `design-review` are unchanged.

15. The spec does not define rollback behavior if modularization makes a skill worse or breaks invocation.
Fix: Require preserving the old monolithic content in git history plus a quick revert path, or implement one skill first and validate before splitting all three.

VERDICT: REVISE
## Round 2 — Codex
**Prior Findings Check**

Most round-1 issues are addressed: no fake `AskUserQuestion` dependency, full step mapping is present, `ISSUE-N.md` vs PRD granularity is separated, `writing-plans` gets an explicit override, and the PRD what/how boundary is much better bounded.

**Remaining / New Findings**

1. The “structurally faithful” verification rule contradicts Items 2-4. It says the only permitted differences are Reads/Hands-off framing, but the spec also intentionally changes interview wording, issue granularity flow, and PRD scope rules.
Fix: Change the diff rule to allow “Reads/Hands-off framing plus explicitly listed behavioral edits from Items 2-4.”

2. `design-brainstorm` says “Add a third override,” but the current file already has three overrides: terminal state, success criteria, and spec writing.
Fix: Say “Add another override” or “Add OVERRIDE — default selection,” without numbering it as third.

3. Thin `SKILL.md` files keeping old `Hard Rules` may leave stale references like “Do NOT proceed past Step 5,” after the process is replaced by step files.
Fix: Require updating internal step-number references to the new step-file names, while preserving the behavioral rule.

4. The staged rollout says validate `discover` before continuing, but does not define who decides it passed or what “successful dry-run” means.
Fix: Define pass criteria: all setup/checkpoint/session/resume/write/commit/handoff steps are exercised, and the user explicitly approves continuing to `ideate`/`prd`.

5. The rollback reference “branch’s parent commit” is fragile once the staged rollout creates intermediate commits; implementers may diff against the wrong parent.
Fix: Record the base SHA before the first split and use that SHA for all structural diff comparisons.

6. `plan`’s “Pass the user's choice into writing-plans as a hard constraint” is still slightly magical because `writing-plans` is an invoked skill, not a callable API with typed parameters.
Fix: Specify that the chosen granularity must be included verbatim in the override text handed to `superpowers:writing-plans`.

7. The default-confirmation examples use “Confirm or tell me what to change,” but `discover` currently rejects vague confirmations like “sounds good.”
Fix: State that explicit confirmation requirements still apply; “confirm” means a clear yes/choice, not passive agreement.

These are implementation-spec cleanup issues, not fundamental design blockers. The spec is sound enough to proceed once those wording fixes are folded into planning.
